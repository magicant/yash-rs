# Job control

**Job control** is a facility that allows users selectively to stop (suspend) the execution of processes and continue (resume) their execution at a later point. It also allows users to interrupt running processes.

## Why is job control useful?

Let's start with a simple example. Suppose you have started a command to download a file, but you realize that the network is down and want to cancel the download. With job control, you can interrupt the command by pressing the `intr` key sequence (usually `Ctrl-C`), which sends a `SIGINT` signal to the foreground process, terminating it.

```shell,no_run
$ curl 'http://example.com/largefile.zip' > largefile.zip
^C
$
```

Job control allows you to move processes between the foreground and background, giving you flexibility in managing multiple tasks in the shell. Suppose you are running a command that removes a large number of files:

<!-- markdownlint-disable MD014 -->
```shell,no_run
$ rm -rf ~/.my_cache_files
```
<!-- markdownlint-enable MD014 -->

You realize that the command is taking too long, and you wish you had started it in the background so you could continue using the shell. With job control, you can suspend the command running in the foreground and then resume it in the background. To do so, you press the `susp` key sequence (usually `Ctrl-Z`), which sends a `SIGTSTP` signal to the foreground process, suspending it. The shell then displays a message indicating that the job has been stopped:

```shell,no_run
^Z[1] + Stopped(SIGTSTP)     rm -rf ~/.my_cache_files
```

You can then resume the job in the background by typing:

```shell,no_run
$ bg
[1] rm -rf ~/.my_cache_files
```

Now the command continues running in the background, and you can continue using the shell for other tasks. If you want to bring the job back to the foreground, you can use the `fg` command:

```shell,no_run
$ fg
rm -rf ~/.my_cache_files
```

Another useful scenario is when you use an editor to edit a file, but you want to run a command in the shell while the editor is open. You can suspend the editor with `susp` (`Ctrl-Z`), run the command, and then return to the editor with `fg`.

## How does job control work?

When the shell starts a [subshell](../environment/index.html#subshells), it creates a new process group for that subshell. If the subshell is run synchronously (in the foreground), the shell makes the subshell the foreground process group of the terminal. The foreground process group receives signals generated by key sequences like `intr` (interrupt) and `susp` (suspend). When a process in the foreground receives a `SIGINT` signal, it is typically terminated. When it receives a `SIGTSTP` signal, it is suspended.

When a foreground process is suspended, the shell displays a message indicating that the job has been stopped, and continues to read a next command. The shell maintains a list of jobs so that you can manage them later. When you use the `fg` built-in, the shell makes the specified job the foreground process group again and sends it a `SIGCONT` signal to resume its execution. If you use the `bg` built-in, the shell just sends the job a `SIGCONT` signal, allowing it to continue running in the background.

All commands in a [pipeline](../language/commands/pipelines.md) run in the same process group. This allows you to manage the entire pipeline as a single job.

When the shell is reading input, it makes itself the foreground process group of the terminal. This means that key sequences like `intr` and `susp` send signals to the shell itself. However, the shell configures itself to ignore these signals, so that the job control shell is protected from being interrupted or suspended unintentionally.
